# Quickstart: AR Internal World Inside Image Marker

**Feature**: 001-ar-internal-world
**Date**: 2026-05-07

This is the fastest path from a fresh checkout to seeing the contained miniature world inside the marker on a real phone. It targets the v1 implementation produced from this plan.

---

## 1. Prerequisites

- A printed copy of the marker image (the same image used to compile `assets/targets/marker.mind`). A high-quality printout (laser/inkjet, matte preferred) ~10 cm wide is plenty.
- A current mobile phone:
  - iOS: Safari on iOS 15+
  - Android: Chrome (recent)
- A laptop on the same Wi-Fi network as the phone.
- Either:
  - **`mkcert`** installed locally (for self-signed HTTPS), **or**
  - **`ngrok`** account / CLI (for a public HTTPS tunnel).

The experience requires HTTPS to access the camera on a non-localhost origin. There is no way around this — it is a browser security policy, not a project choice.

## 3. Compiling the marker target file (one-time)

If you do not already have `marker.mind`:

1. Open MindAR's official online image-target compiler (part of the `mind-ar-js` documentation site — search "MindAR image target compiler").
2. Upload the source image.
3. Wait for compilation; verify the tracking-quality score is **≥ 4 / 5**. If lower, choose a more feature-rich source image — flat-color graphics track poorly.
4. Download the resulting `.mind` file and save it as `assets/targets/marker.mind`.
5. Print the **same** image. Compiled target and printed image must match.

This step is one-time per marker design. Recompile only if you change the marker image.

## 4. Running locally — option A: mkcert + http-server

```bash
# 1. one-time: install local CA
mkcert -install

# 2. from project root, generate certs for your laptop's LAN IP
mkcert <your-lan-ip>            # produces cert.pem + key.pem in cwd

# 3. serve with HTTPS on port 8080
npx http-server -S -C cert.pem -K key.pem -p 8080
```

On the phone:

1. Connect to the same Wi-Fi as the laptop.
2. Open `https://<your-lan-ip>:8080` in Safari (iOS) or Chrome (Android).
3. Accept the certificate warning (mkcert-issued certs are not trusted on phones unless you import the root CA).
4. Grant camera permission when prompted.

If the certificate warning is a blocker on iOS, fall back to option B (ngrok).

---

## 5. Running locally — option B: ngrok tunnel

```bash
# 1. plain http-server on port 8080
npx http-server -p 8080

# 2. in another terminal, tunnel it
ngrok http 8080
```

Ngrok prints an `https://*.ngrok-free.app` (or similar) URL.

On the phone:

1. Open the printed `https://*.ngrok-free.app` URL.
2. Grant camera permission when prompted.

## 6. Expected first-run behavior

1. Page loads. Loading overlay visible.
2. A-Frame and MindAR finish loading; `marker.mind` and `human_head.glb` finish fetching.
3. Camera permission prompt appears. Grant.
4. Loading overlay disappears. "Point the camera at the marker" hint appears.
5. Aim the phone at the printed marker.
6. Within ~2 seconds, the colored interior chamber + centered head fade in inside the marker. The hint disappears.
7. Move the phone or tilt the marker — the world stays anchored to the marker; walls slide past, the head occludes the back wall, the front opening foreshortens at oblique angles.
8. Move the marker out of frame — the world fades out cleanly within ~1 second.
9. Move the marker back in — the world fades back in within ~2 seconds.

If any of these steps fails, see Troubleshooting below.

## 7. Manual QA checklist (per-feature spec acceptance)

Run through this before declaring an iteration done. Each item maps to one or more spec FR/SC items.

- [ ] **Detection (US1, FR-001, SC-001)**: World appears within ~2 s of marker entering frame, on first attempt, ≥ 9/10 tries under indoor lighting.
- [ ] **Color mapping (FR-007)**: Right wall = red, left = green, ceiling = blue, floor = yellow, back = purple. Front-opening rim is visibly orange.
- [ ] **Inward-facing surfaces (FR-006, FR-008)**: From inside the chamber, all six surfaces (or rim) are visible. From a steep oblique angle, only the chamber interior is visible — never the back of a wall.
- [ ] **Head centered (FR-009)**: Head sits at chamber center, fits comfortably, faces the front opening.
- [ ] **Marker-anchored (FR-002, FR-013)**: Rotating the marker in plane rotates the chamber identically.
- [ ] **Parallax (US2, FR-011, FR-012, SC-003)**: Tilt left ↔ side walls reveal/recede; tilt up/down ↔ ceiling/floor reveal; move close ↔ chamber foreshortens; head visibly occludes back wall from side angles.
- [ ] **Stable on hold (FR-005, SC-002)**: With marker held still, no perceptible jitter for ≥ 30 s.
- [ ] **Fade in (FR-018)**: First reveal is a smooth fade, not a pop-in.
- [ ] **Fade out (FR-019)**: Marker out-of-frame ⇒ smooth fade-out within ~1 s; no residual geometry.
- [ ] **Re-acquisition (US3, FR-004, SC-010)**: Marker back in frame ⇒ world fades back in within ~2 s.
- [ ] **Flicker tolerance (FR-020, SC-010)**: Rapidly cover and uncover marker ~5 times in 15 s — the world does not strobe; transitions stay smooth.
- [ ] **Camera permission denied (FR-023, SC-008)**: Deny permission once. Clear message + retry instructions appear. No blank screen.
- [ ] **Asset error (FR-027, SC-009)**: Temporarily rename `human_head.glb`. Reload. Asset-error overlay appears. Restore file; reload; works again.
- [ ] **Aim-at-marker hint (FR-025)**: First load, with marker not yet in frame, the "point the camera at the marker" hint is visible. Disappears on first detection.
- [ ] **Loading state (FR-026)**: On first load (cache cleared), a loading indicator is visible until the world is ready to render.
- [ ] **Cross-platform (FR-021, SC-006)**: Above flow works on both iOS Safari and Android Chrome on the configured test phones.
- [ ] **Smooth motion (FR-028, SC-007)**: No perceptible stutter during natural viewing motion on a current mid-range phone.

## 8. Tunable defaults (where to adjust)

If the visual feel is off in QA, the cheapest knobs to turn are at the top of `js/app.js` and in component schema defaults:

| Knob | Default | Effect |
|---|---|---|
| Chamber width / height / depth | 1.0 / 1.0 / 1.0 | Larger depth → stronger parallax; mismatched dims → boxier or longer chamber |
| Head scale | 0.4 | Smaller → more space around the head; larger → more imposing presence |
| Head Y rotation | 180° | Face direction toward viewer |
| Fade-in / fade-out duration | 350 ms | Snappier vs. more cinematic transitions |
| Tracking-loss grace window | 120 ms | Larger → smoother through flickers, but lingers longer when truly lost |
| Jitter smoother α | 0.3 | Higher → more responsive but more jitter; lower → smoother but laggier |
| Ambient intensity | 0.45 | Higher → flatter, less depth; lower → moodier |
| Directional intensity | 0.9 | Shapes face geometry; tune with directional position |
| Rim accent intensity | 0.4 (or off) | Silhouettes head against back wall |

## 9. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Page won't load on phone over `https://<lan-ip>` | Cert not trusted by phone | Use ngrok (option B) or import mkcert root CA on the phone |
| Camera permission prompt never appears | HTTPS not in use, or browser doesn't support `getUserMedia` | Confirm URL is `https://`; switch to a supported browser |
| World never appears even with marker in frame | Marker tracking quality too low, or `marker.mind` mismatched with printed image | Recompile target with a higher-quality source; reprint at higher resolution; ensure printed image matches compiled image |
| World appears but jitters heavily on stationary marker | Jitter smoother α too high, or marker has very low tracking quality | Lower α toward 0.2; recompile target with richer source image |
| World appears outside the marker, floating in space | World is parented to the wrong entity (not `mindar-image-target`) | Code regression — re-check entity hierarchy; chamber must be a descendant of `<a-entity mindar-image-target>` |
| Walls flicker or appear backward (back of plane visible) | `material.side` not set to `back`, or normal/rotation incorrect | Ensure each chamber surface uses `material="side: back"` and that its position/rotation matches the canonical mapping in data-model.md |
| World fades in but head missing | GLB load error not surfacing, or scale/origin off so the head is outside the visible frustum | Check console for `model-error`; verify the head's origin sits inside the chamber after scale is applied |
| Strobing reveal/hide on partial occlusion | Grace timer too short, or `tracking-fade` not subscribed correctly | Increase grace to ~180 ms; verify only one component subscribes to `targetFound`/`targetLost` |
| Frame rate drops on mid-range phone | Head poly count or texture size over budget, rim light enabled on a low-tier device | Disable rim light; reduce texture resolution; LOD-swap head model |
